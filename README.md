## 1. 產品概述

### 1.1 產品定義
一套綜合的反爬蟲防護系統，保護電商平台的商品列表頁面免受自動化爬蟲訪問，通過多層驗證機制確保只有正常用戶能訪問敏感內容。

### 1.2 核心目標
- **安全性**: 阻攔爬蟲和異常流量
- **用戶體驗**: 合法用戶無感知，驗證流程簡單快速
- **可維護性**: 易於配置和監控

### 1.3 適用範圍
- 電商商品列表頁面 (`/products`)
- Google reCAPTCHA v2 集成驗證

---

## 2. 目標用戶

| 角色 | 描述 | 需求 |
|------|------|------|
| **正常用戶** | 瀏覽商品的合法客戶 | 快速訪問，無過度驗證 |
| **爬蟲使用者** | 試圖自動化抓取數據 | 被有效阻攔 |
| **平台管理員** | 系統管理和監控 | 實時查看違規 IP、調整策略 |

## 3. 功能需求

### 3.1 L1 - 核心功能

#### 3.1.1 User-Agent 檢測
- **功能**: 識別可疑的 User-Agent 字串
- **黑名單詞彙**:
  - `python-requests`, `guzzlehttp`, `scrapy`
  - `headlesschrome`, `selenium`, `puppeteer`
  - `curl`, `wget`, `postman`
- **觸發條件**: UA 為空或包含黑名單關鍵字
- **處理**: 跳轉到驗證頁面

#### 3.1.2 Google reCAPTCHA v2 驗證
- **驗證方式**: Google reCAPTCHA v2（用戶勾選「我不是機器人」）
- **後台驗證**: 伺服器端驗證 reCAPTCHA token
- **成功條件**: Google 返回 success: true
- **失敗處理**: 
  - 失敗 1-2 次: 提示剩餘次數，重置驗證框
  - 失敗 3 次: IP 被 block 1 小時

#### 3.1.3 頻率限制
- **限制規則**: 同一 IP 每分鐘最多 50 次請求
- **超限處理**: 觸發驗證碼驗證
- **時間窗口**: 分鐘級別（HH:mm 格式）

#### 3.1.4 IP 狀態管理
- **狀態類型**:
  - `pending_captcha`: 等待驗證
  - `blocked`: IP 已被暫時封鎖
  - `verified`: IP 已通過驗證
- **儲存位置**: Redis
- **過期時間**: 1 小時

#### 3.1.5 驗證成功後處理
- 清除 IP 對應的 Redis 狀態
- 清除驗證失敗計數
- 重新加載原始頁面
- 1 小時內不再要求驗證

### 3.2 L2 - 增強功能

#### 3.2.1 詳細錯誤提示
- 驗證失敗時顯示剩餘嘗試次數
- IP 被 block 時提示解封時間
- AJAX 錯誤捕獲並展示後台消息

#### 3.2.2 日誌記錄
- 記錄所有可疑訪問
- 記錄驗證失敗的 IP
- 記錄被 block 的 IP 和原因

#### 3.2.3 管理員監控面板 (未實現)
- 實時查看被 block 的 IP 列表
- 驗證失敗統計
- 手動解除 block 功能

## 5. 使用者流程

### 5.1 正常用戶流程
```
訪問 /products 
  ↓
User-Agent 正常
  ↓
頻率限制未超
  ↓
Redis 無 block 狀態
  ↓
✅ 直接加載商品列表
```

### 5.2 可疑流量流程
```
訪問 /products
  ↓
User-Agent 可疑 OR 頻率超限
  ↓
設置 pending_captcha 狀態
  ↓
🔐 重定向到驗證頁面
  ↓
用戶完成 reCAPTCHA
  ↓
提交驗證
  ↓
成功 → 清除狀態 → 重新加載
失敗 → 增加失敗計數
失敗3次 → 設置 blocked 狀態 → 1小時內拒絕訪問
```

### 5.3 爬蟲阻攔流程
```
爬蟲自動訪問 (User-Agent: python-requests)
  ↓
檢測到黑名單 UA
  ↓
設置 pending_captcha
  ↓
爬蟲無法處理驗證頁面
  ↓
❌ 爬蟲停止
```

## 6. 系統架構

### 6.1 核心組件

```
User Request
    ↓
[web.php] 路由定義
    ↓
[AntiBotMiddleware] 檢查流程
    ├─ UA 檢測
    ├─ 頻率限制
    ├─ Redis 狀態檢查
    └─ 決策（允許/驗證/拒絕）
    ↓
[Frontend] 驗證頁面
    ├─ 顯示 reCAPTCHA v2 Checkbox
    ├─ 用戶勾選驗證
    └─ 提交表單
    ↓
[CaptchaController] 驗證邏輯
    ├─ 解析 reCAPTCHA response
    ├─ 調用 Google API
    ├─ 失敗計數
    ├─ Block 決策
    └─ 返回 JSON 結果
    ↓
[Frontend JavaScript]
    ├─ 處理驗證結果
    ├─ 顯示錯誤信息
    ├─ 重置驗證框
    └─ 重新加載或重試
    ↓
Redis (狀態存儲)
    ├─ bot_status:{ip}
    ├─ captcha_fail:{ip}
    └─ rate_limit:{ip}:{time}
```

### 6.2 資料流

| 數據 | 存儲位置 | TTL | 說明 |
|------|---------|-----|------|
| bot_status:{ip} | Redis | 1h | IP 的驗證狀態 |
| captcha_fail:{ip} | Redis | 1h | 驗證失敗計數 |
| rate_limit:{ip}:{time} | Redis | 1m | 分鐘級請求計數 |

## 7. 技術規格

### 7.1 後端技術棧
- **框架**: Laravel 10
- **中間件**: AntiBotMiddleware
- **驗證API**: Google reCAPTCHA v2
- **緩存**: Redis
- **HTTP Client**: Guzzle (Laravel Http Facade)

### 7.2 前端技術棧
- **JavaScript**: jQuery 3.7.1

```env
RECAPTCHA_SECRET_KEY=your_secret_key
REDIS_HOST=127.0.0.1
REDIS_PORT=6379
```

## 8. 功能規格詳細定義

### 8.1 驗證失敗策略

| 失敗次數 | 行為 | 提示信息 |
|---------|------|--------|
| 1 | 增加計數 | 「驗證失敗，還有 2 次機會」 |
| 2 | 增加計數 | 「驗證失敗，還有 1 次機會」 |
| 3 | 設置 block | 「驗證失敗次數過多，IP 已被封鎖 1 小時」 |
| 4+ | 403 Forbidden | 「您的 IP 因驗證失敗多次已被暫時封鎖，1 小時後重試」 |

### 8.2 API 端點規格

#### POST /captcha/verify
**請求參數:**
```json
{
  "g-recaptcha-response": "string",
  "_token": "csrf_token"
}
```

**成功響應 (200):**
```json
{
  "success": true
}
```

**失敗響應 (422):**
```json
{
  "success": false,
  "message": "驗證失敗，還有 2 次機會"
}
```

## 10. 風險分析與對策

### 10.1 已知風險

| 風險 | 影響 | 對策 |
|------|------|------|

## 12. 常見問題 (FAQ)

**Q: 正常用戶會被誤判嗎?**  
A: 極少發生。只有在使用特定爬蟲工具或頻率異常時才會觸發驗證。

**Q: 驗證成功後需要重複驗證嗎?**  
A: 不需要。同一 IP 在 1 小時內驗證成功後不會再被要求驗證。

**Q: 如何解除 block?**  
A: 自動解除。Block 狀態 1 小時後 Redis 過期，用戶可重新驗證。

**Q: 爬蟲可以繞過驗證嗎?**  
A: 完整繞過困難，但仍可能被有經驗的攻擊者利用。這是多層防護的第一層。
